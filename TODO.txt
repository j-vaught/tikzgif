ROBUSTNESS
================================================================================

[ ] Test Suite
    Write unit tests for template parsing, config validation, bounding box
    extraction, cache store operations, and color quantization. Write integration
    tests for the full render pipeline using small 3-frame examples. Write edge
    case tests for bad .tex input, missing \PARAM token, single frame renders,
    missing LaTeX engines, and unavailable rasterization backends. Target the 80%
    coverage threshold already configured in pyproject.toml.
    WHY: No tests exist. This is the single largest gap in the project. Without
    tests, refactoring is dangerous, regressions go unnoticed, and no serious
    user or contributor will trust the tool. JOSS submission also requires tests.

[ ] CI/CD Pipeline
    Set up GitHub Actions to run pytest, mypy (strict), and ruff on every push.
    Test matrix should cover Python 3.10-3.13 on Ubuntu and macOS. Add status
    badges to the README.
    WHY: Automated testing catches breakage before it ships. Badges in the README
    signal project health to potential users and contributors.

[ ] Remove Unused Optional Dependencies
    The cairosvg (pdf2svg group) and watchdog (watch group) optional dependencies
    are declared in pyproject.toml but nothing in the codebase uses them. Either
    implement the features they support or remove the declarations.
    WHY: Dead dependency declarations confuse contributors and create false
    expectations about what the tool can do.

[ ] Actionable Error Messages
    Ensure all error messages include enough context for the user (or an AI agent)
    to fix the problem without reading source code. LaTeX compilation errors
    should identify the missing package name and suggest the install command
    (e.g., tlmgr install pgfplots). Rasterization errors should name the missing
    backend and suggest platform-specific install commands. Include the failing
    frame number, source line, and relevant file paths.
    WHY: Vague error messages ("compilation failed") force users into manual
    debugging. Specific messages enable self-service resolution and allow AI
    agents to self-heal in automated workflows.

[ ] Deterministic Output
    Ensure identical input always produces byte-identical output when using the
    same parameters. Audit GIF assembly for sources of nondeterminism (metadata
    timestamps, dict ordering, floating point rounding in color quantization).
    WHY: Determinism enables reproducible research figures, meaningful diffing of
    outputs in version control, and predictable behavior for automated pipelines.

[ ] Benchmark Suite
    Create a small set of benchmark examples (e.g., 120-frame Lorenz attractor)
    with a script that measures wall-clock time for sequential vs. parallel
    compilation, cached vs. uncached runs, and different rasterization backends.
    Record results in a reproducible format.
    WHY: Quantitative performance claims (for the README, JOSS paper, and
    marketing) need to be backed by reproducible measurements. Also useful for
    catching performance regressions.

================================================================================
MACHINE USABILITY
================================================================================

[ ] --json Flag on All Commands
    Add a --json flag to render, inspect engines, inspect backends, and inspect
    template. When set, suppress human-readable output (progress bars, emoji,
    status messages) and print a single JSON object to stdout with structured
    results. The render command should return: status, output_path, frames_compiled,
    frames_cached, frames_failed, duration_seconds, output_size_bytes, and
    output dimensions.
    WHY: AI agents and automated pipelines cannot parse tqdm progress bars or
    human-readable text. JSON output is the standard interface for machine
    consumption. This is the single highest-impact change for agent adoption.

[ ] Structured Exit Codes
    Define and document specific exit codes beyond 0/1. Suggested mapping:
      0 = success
      1 = LaTeX compilation error (fixable input)
      2 = missing system dependency (agent can install)
      3 = bad input file or template error (agent can fix the .tex)
      4 = output write failure (permissions, disk space)
    WHY: Agents use exit codes to decide their next action. A single nonzero
    exit code forces the agent to parse stderr to figure out what went wrong.
    Structured codes enable immediate branching.

[ ] Stderr/Stdout Separation
    Route all progress bars, status messages, warnings, and logs to stderr.
    Route only parseable output (JSON, file paths) to stdout. Add a --quiet
    flag that suppresses stderr entirely. Add --output-path-only that prints
    just the output file path to stdout.
    WHY: Agents and scripts pipe stdout. If progress bars and output paths are
    mixed on stdout, piping breaks. Clean separation enables:
    tikzgif render input.tex --quiet --output-path-only | xargs open

[ ] Stdin Support
    Accept .tex source from stdin via a --stdin flag:
      cat input.tex | tikzgif render --stdin --output out.gif
    WHY: AI agents often generate .tex code in memory and prefer piping it
    directly rather than writing a temp file first. Reduces friction in
    automated workflows.

[ ] MCP Server
    Implement a Model Context Protocol server that exposes tikzgif_render,
    tikzgif_inspect_engines, and tikzgif_inspect_backends as MCP tools. Each
    tool should accept structured parameters and return structured results.
    Package as a standalone entry point (tikzgif-mcp) or integrate into the
    main CLI (tikzgif mcp serve).
    WHY: MCP is how Claude Code, Cursor, Windsurf, and other AI coding tools
    discover and call external tools natively. An MCP server lets any
    MCP-compatible agent render TikZ animations without knowing CLI flags.
    This is the highest-leverage integration point for AI agent adoption.

[ ] --help-json Flag
    Add a flag that outputs the full CLI schema (all commands, all flags, types,
    defaults, descriptions) as a JSON object.
    WHY: Agents that encounter tikzgif for the first time can call --help-json
    once and immediately understand every available option without parsing
    human-readable help text.

[ ] Python API with Structured Returns
    Ensure the public Python API (tikzgif.render, tikzgif.render_job) returns
    structured result objects (not just writing files as a side effect). The
    result should include output_path, frame_count, cached_count, failed_count,
    duration, and output metadata. Document the API with clear type hints and
    docstrings.
    WHY: AI agents running in Python prefer importing a library over shelling
    out to a subprocess. A clean programmatic API with typed return values is
    the most ergonomic interface for Python-based agents.

[ ] --watch Mode
    Implement file watching using the watchdog dependency (already declared).
    When a .tex file changes, automatically re-render the animation. Output
    the path to the new file on each re-render.
    WHY: Enables a live-preview workflow for iterative animation development.
    Also useful for agents that modify .tex files and want to see results
    without re-invoking the CLI.

[ ] Jupyter Cell Magic (%%tikzgif)
    Create an IPython cell magic that renders TikZ source inline in a Jupyter
    notebook and displays the resulting GIF in the cell output.
    WHY: Academics and data scientists live in Jupyter notebooks. A cell magic
    makes tikzgif accessible to this large audience without leaving their
    workflow. Also usable by AI agents operating in notebook environments.
